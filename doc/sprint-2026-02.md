### 3. Reader View (Feb 8)

Added a second tab (`#/reader`) that renders article HTML received from the ReadEasy Chrome extension via `window.postMessage`.

**What it does:**
- Listens for `postMessage` events from two known Chrome extension IDs
- Validates origin strictly (reject all other origins)
- Sanitizes incoming HTML: removes `<script>`, `<iframe>`, event handlers, `javascript:` URLs; resolves relative URLs against `sourceUrl`
- Displays article title, byline, source link, and sanitized body in a clean reading layout
- Sends `readeasy-ready` handshake to `window.opener` so the extension can deliver payload immediately

**Reader toolbar:**
- Theme toggle: dark ↔ light (dark is default)
- Font size presets: 16, 18, 20, 22 px

**Responsive layout:**
- Reader content capped at readable max-width
- Navigation adjusts for mobile

**Handshake flow:**
```
Extension opens /#/reader in new tab
       ↓
Web app sends 'readeasy-ready' to window.opener
       ↓
Extension sends { type: 'readeasy-article', title, byline, siteName, sourceUrl, html }
       ↓
Web app validates origin, sanitizes HTML, renders content
```

### 4. EPUB Download from Reader (Feb 13)

Added a "Download EPUB" button to the reader toolbar. Builds a valid EPUB 2.0 file entirely in the browser using **JSZip**.

**Process:**
1. Parse article HTML, find `<img>` tags with `http(s)://` sources
2. `fetch()` each image and embed in ZIP at `OEBPS/images/img{n}.{ext}`
3. Build EPUB structure: `mimetype`, `META-INF/container.xml`, `OEBPS/content.opf`, `OEBPS/toc.ncx`, `OEBPS/content.xhtml`
4. Run XHTML compliance pass: self-close void elements (`<br />`, `<img />`), remove event handlers, escape `&nbsp;` → `&#160;`, escape bare `&`
5. Generate ZIP blob, trigger browser download with sanitized filename

**Key decisions:**
- Use browser-native `fetch()` for images — fails gracefully if an image is unreachable (logs warning, keeps original URL or omits image)
- Apple Books compatibility: required `toc.ncx` in addition to inline TOC
- Button is disabled until an article is loaded to prevent accidental empty downloads

**Bug fixes during development:**
- Fixed void element self-closing (especially `<source>`)
- Fixed attribute sanitization that was producing malformed XHTML (`//> ` artifacts)
- Fixed syntax error in `convertToXhtml`
- Added detailed `postMessage` logging to debug extension integration
- Added automatic navigation to `#/reader` when article is received (in case user is on merge tab)

### 5. Convert Text Tab (Feb 13)

Added a third tab (`#/convert`) where users can paste plain text and preview it in reader mode.

**What it does:**
1. User types/pastes text into a textarea (optional title field)
2. `convertPlainTextToHtml()` escapes HTML entities, splits on double newlines, wraps in `<p>` tags
3. Produces a `readeasy-article` payload and calls `renderReaderContent()` directly (no network, no extension)
4. Navigates to `#/reader` to display result
5. Article is available for EPUB download via the existing reader toolbar button

**State persistence:**
- `readerState.convertText` and `readerState.convertTitle` store current input values so they survive tab switches

### 6. Google Authentication (Feb 13)

Gated the Convert Text tab behind Google Sign-In.

**Implementation:**
- Uses **Google Identity Services (GSI)** — the Google-hosted library renders the sign-in button
- `setupGoogleAuth()` initializes GSI with the project's OAuth client ID, with retry logic (up to 10 × 300ms) for script load race conditions
- On credential callback: decode JWT payload with `atob()`, extract `name`, `email`, `picture`
- Persist `{ idToken, profile }` to `localStorage` under key `readeasy-auth`
- Load on startup: `loadAuthState()` restores session from `localStorage`

**Auth-gated routing:**
- Convert tab shows a lock indicator when logged out (`locked` CSS class)
- Clicking the locked tab is a no-op (event prevented)
- Navigating directly to `#/convert` while logged out redirects to `#/reader`
- On logout: clear `localStorage`, redirect to `#/reader`

**UI:**
- Logged out: Google sign-in button visible in nav area
- Logged in: User avatar + name visible; logout button appears; sign-in button hides

**Note:** The `idToken` is stored but not currently validated server-side. It is used only as a client-side session signal.

## Key Decisions

**Why client-side EPUB generation for the reader (not the backend API)?**
The reader receives article HTML from an extension — there is no EPUB file to send. Building EPUB in-browser with JSZip avoids a round-trip to the server, works offline, and keeps the feature simple.

**Why Google Sign-In for the convert feature?**
The convert tab is a higher-value feature that may eventually feed into a backend (saving articles, history, etc.). Google login provides a real identity hook with no password management overhead.

**Why vanilla JWT decode (no library)?**
The `idToken` is only used to show the user's name and avatar client-side. A full verification library would be overkill. Server-side validation of the token should be added when a backend feature needs it.

**Why hash routing instead of a router library?**
The app has three views. Hash routing is 10 lines of code; a library would be disproportionate.
