## What We Built

### EPUB Combiner UI

A single-page application built with HTML, CSS, and vanilla JavaScript — no framework, no build step. The app lets users upload 2–10 EPUB files and combine them into one via a backend API.

**Core features shipped:**
- Drag-and-drop upload area with visual feedback (border highlight on drag)
- File validation: `.epub` extension only, 50 MB per file, max 10 files, 100 MB total
- File list with individual remove buttons
- Progress bar with four staged updates (30% → 60% → 90% → 100%)
- Auto-download of the combined EPUB on success
- Error display with 5-second auto-dismiss and a retry button
- Reset flow to combine another batch without page reload
- Configurable API URL with "Test Connection" button that fetches `/config` and shows limits

**UI state machine:**
```
IDLE → UPLOADING (files selected) → COMBINING (progress bar) → SUCCESS / ERROR
```

**Responsive design:**
- CSS variables for all colors (easy re-theming)
- Single breakpoint at 640px for mobile

### Backend API

Express.js server deployed to Render. Accepts multipart file uploads, unzips each EPUB, merges content and images (namespace-prefixed to avoid collisions), generates a TOC page, repacks into a new EPUB, and streams the binary back.

**Endpoints:**
- `GET /health` — liveness check
- `GET /config` — returns upload limits
- `POST /combine-epubs` — the main operation

### Deployment

| Layer | Hosting | URL |
|-------|---------|-----|
| Frontend | Vercel | https://merge-epubs.vercel.app |
| Backend | Render (free tier) | https://epub-combiner-api.onrender.com |

Initial architecture had a Vercel serverless proxy relaying requests to Render. This was removed by end of sprint (see January 31–February 2 work) to reduce latency and eliminate the extra hop.

---

## Key Decisions

**Why vanilla JS (no React/Vue)?**
The app is a single operation with a small amount of state. A framework would add build complexity with no benefit at this scale.

**Why Vercel for frontend?**
Zero config for static sites, automatic HTTPS, and instant deploys on git push.

**Why Render for backend?**
Free tier is sufficient for the expected usage. The API is stateless (no database), so no data persistence concerns on the free tier.

**Why remove the Vercel proxy?**
The proxy added latency and the Vercel free tier has a 10-second function timeout — not enough for large EPUB processing jobs. Direct browser → Render API calls have a 5-minute timeout and lower overhead.
